1. 이 라이브러리는 무엇인가?

JobDispatcherNET은 여러 스레드에서 작업을 동시에 처리하면서도, lock을 거의 사용하지 않고 안전하게 데이터를 다룰 수 있게 해주는 라이브러리입니다.

핵심 아이디어: 각 객체가 자기만의 "우체통(Queue)"을 가집니다.
외부에서는 우체통에 편지(작업)를 넣기만 하고, 실제 읽기는 한 번에 하나의 스레드만 합니다.
→ 같은 우체통의 편지는 절대 동시에 열리지 않으므로 lock이 필요 없습니다.

2. 어떤 문제를 해결하는가?

❌ 일반적인 멀티스레드

여러 스레드가 같은 데이터에 접근 → lock 필요

lock (_playerLock)
{
    player.Hp -= damage;  // 안전하지만 느림
}

문제: lock이 많아지면 스레드끼리 서로 기다리며 성능이 떨어집니다.

✅ JobDispatcherNET

작업을 큐에 넣으면 알아서 직렬 처리 → lock 불필요

player.DoAsync(() =>
{
    player.Hp -= damage;  // lock 없이 안전!
});

해결: 같은 객체의 작업은 자동으로 순서대로 실행됩니다.

3. 핵심 클래스 6개

각 클래스의 역할 요약
  • AsyncExecutable — 상속받는 클래스에 "자기만의 작업 큐"를 부여합니다. 가장 중요한 클래스.
  • JobDispatcher<T> — 전용 OS 스레드를 N개 만들어 워커를 실행합니다.
  • IRunnable — 워커 스레드의 메인 루프를 정의하는 인터페이스.
  • Job — Action을 감싸는 작업 단위. 오브젝트 풀로 재사용됩니다.
  • TimerQueue — 일정 시간 후 작업 실행을 예약합니다.
  • ThreadContext — 스레드별 저장소. 각 워커 스레드의 타이머, 큐 등을 관리합니다.

5. DoTask 내부 동작

DoTask()는 AsyncExecutable의 심장입니다. 3가지 시나리오를 단계별 애니메이션으로 살펴보세요.

단계를 시작하려면 ▶ 다음을 누르세요
핵심 포인트 정리:
  • 시나리오 A — count가 0→1이면 이 스레드가 "주인"이 되어 직접 Flush합니다.
  • 시나리오 B — count > 1이면 큐에만 넣고 즉시 돌아갑니다. 기다리는 시간 = 0. 이미 Flush 중인 스레드가 대신 처리.
  • 시나리오 C — Flush 중에 다른 객체의 DoAsync가 호출되면 ExecuterQueue에 등록하고, 현재 Flush가 끝난 후 이어서 처리합니다.

→ 어떤 시나리오에서도 lock이나 대기가 없습니다. 이것이 JobDispatcherNET의 핵심입니다.

6. 타이머 (DoAsyncAfter)

TimerQueue 동작 원리
  1. DoAsyncAfter(5초, action) 호출 → PriorityQueue에 (dueTime, job) 저장
  2. 백그라운드 PeriodicTimer가 1ms마다 깨어나서 만료된 작업을 확인
  3. 시간이 되면 owner.DoTask(job) 호출 → 해당 AsyncExecutable의 큐로 전달
  4. 이후는 일반 DoAsync와 동일하게 워커 스레드에서 처리

Stopwatch 기반이므로 DateTime.UtcNow(~15ms)보다 훨씬 정밀합니다.

7. 실전 패턴 — MMORPG 서버

아래는 존이 1개인 MMORPG에서 플레이어 Actor 패턴을 적용한 예시입니다.

// 플레이어 Actor 정의 — AsyncExecutable 상속
public class PlayerActor : AsyncExecutable
{
    private readonly Player _player;

    // 이동 — 자기 Actor에서 실행, 완전 병렬
    public void Move(float newX, float newY)
    {
        DoAsync(() =>
        {
            _player.X = newX;      // lock 없이 안전!
            _player.Y = newY;      // 이 Actor의 큐에서만 실행되니까
        });
    }

    // 근접 공격 — 스냅샷 캡처 후 대상 Actor로 전달
    public void MeleeAttack(PlayerActor target)
    {
        DoAsync(() =>
        {
            // 1단계: 내 상태를 스냅샷으로 복사 (불변)
            var snapshot = new { _player.X, _player.Y, _player.Attack };

            // 2단계: 대상 Actor의 큐에 데미지 전달
            target.ReceiveDamage(snapshot.X, snapshot.Y, snapshot.Attack);
        });
    }
}

8. 한눈에 보는 요약

핵심 규칙 3가지
  1. 같은 Actor의 작업은 자동으로 순서대로 실행됩니다 — lock이 필요 없습니다.
  2. 다른 Actor의 작업은 완전히 병렬 처리됩니다 — 플레이어가 100명이면 100개가 동시에.
  3. Actor 간 통신은 스냅샷으로 — 불변 데이터를 복사해서 전달하면 안전합니다.
⚠️ 주의사항
  • DoAsyncAfter는 워커 스레드의 DoAsync 콜백 안에서 호출해야 타이머가 정상 작동합니다.
  • DisposeAsync() 호출 후에는 DoAsync()를 호출하지 마세요.
  • 에러 핸들링: AsyncExecutable.OnError에 콜백을 등록하면 작업 실행 중 예외를 받을 수 있습니다.

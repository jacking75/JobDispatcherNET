1왜 락(lock) 없이도 안전한가?

멀티스레드 프로그램에서 데이터가 깨지는 이유는, 여러 스레드가 같은 변수에 동시에 쓰기를 하기 때문입니다. 보통은 lock 으로 한 번에 한 스레드만 들어가도록 막죠.

JobDispatcherNET은 다른 길을 갑니다. "하나의 객체에 들어오는 모든 작업을 줄을 세워서 하나씩 처리하자"는 아이디어입니다. 줄(=큐)이 직접 한 번에 하나만 통과시키니, 락이 필요 없어집니다.

이 예제는 전통적 스레드 기반입니다. async/await 키워드를 쓰지 않고, 차단 대기(blocking)에는 ManualResetEventSlim 과 Thread.Sleep 만 사용합니다. 라이브러리 자체는 내부적으로 Task 를 쓰지만, 우리 예제 코드는 종료 시 한 번만 .AsTask().Wait() 로 차단합니다.

그래서 ExampleChatServer의 ChatServer, Room, User 안에는 lock이 단 하나도 없습니다. 모든 진입은 자기 큐를 통과합니다.

2JobDispatcherNET의 3가지 핵심 도구

AsyncExecutable

"내 큐를 가진 객체"가 되는 베이스 클래스. DoAsync(...)로 일을 던지면 그 객체의 큐에 들어가고 한 번에 하나씩 실행됩니다.

JobDispatcher<T> + IRunnable

전용 OS 스레드를 N개 만들어 각 스레드 위에서 T.Run()을 반복 호출합니다. ThreadPool이 아닌 진짜 스레드라서 ThreadLocal 상태가 안전합니다.

DoAsyncAfter

"N초 뒤에 이 코드 실행해줘". 자기 자신을 다시 예약하면 무한히 반복되는 heartbeat가 됩니다.

ThreadContext.TickCount

JobDispatcher 워커가 매 루프마다 갱신하는 ThreadLocal 시각. 워커 스레드에서 도는 코드는 락 없이 시간을 알 수 있습니다.

3ExampleChatServer 전체 지도

이 채팅 서버에는 actor 3종(ChatServer 1개, Room N개, User N개)과 워커 풀(ChatWorker 4개)이 등장합니다. 각 actor는 자기만의 큐를 갖고, 워커는 진짜 OS 스레드 위에서 돕니다.

4Actor 직렬화 — 락이 사라지는 마법

여러 스레드가 동시에 같은 actor의 DoAsync를 호출해도, actor 안의 코드는 한 번에 하나씩 실행됩니다. 그래서 actor 내부 변수는 락 없이도 안전합니다.

핵심 보장: 같은 actor 안에서 도는 코드는 같은 순간에 두 개가 동시에 실행되는 일이 없다. 이게 actor 모델의 전부입니다.
차단(blocking) 호출 주의: GetSnapshot() 같은 동기 read 는 ManualResetEventSlim.Wait() 로 차단합니다. 다른 actor 의 Flush 안에서 부르면 그 스레드가 자기가 기다리는 콜백을 처리하지 못해 데드락이 납니다. 메인 스레드에서만 부르세요. 그래서 ChatServer.GetSnapshot() 은 2단계로 동작합니다 — server 큐에서 참조만 모으고, 호출자 스레드에서 각 actor 의 GetSnapshot() 을 따로 호출합니다.

5명령이 워커를 타고 들어오는 흐름

콘솔에 say alice general 안녕이라고 치면 어떻게 ChatServer까지 도달할까요?

이 흐름이 JobDispatcher<ChatWorker>가 단순 장식이 아니라 진짜로 작업을 처리한다는 증거입니다. 4개 워커가 서로 다른 명령을 동시에 dequeue해서 처리하므로 자연스럽게 병렬화됩니다.

6Room → User 브로드캐스트의 병렬성

Room이 100명에게 채팅 메시지를 뿌릴 때, 만약 Room이 한 명씩 직접 보내며 기다리면 그동안 Room의 큐가 막힙니다. ExampleChatServer는 그렇게 하지 않습니다. Room은 user.DeliverMessage(msg)로 각 User actor의 큐에 작업을 던지고 즉시 다음 사람으로 넘어갑니다. 실제 송신은 각 User actor가 자기 큐에서 처리하므로 병렬화됩니다.

왜 이게 좋은가? 만약 User가 actor가 아니었다면 Room이 4명에게 직접 SendMessage를 호출하면서 Thread.Sleep을 차례로 기다려야 했고, 그동안 Room 큐의 다른 작업(다른 채팅, 입장/퇴장)은 모두 멈췄을 겁니다.

7자기복제 Heartbeat — DoAsyncAfter

"5초마다 통계 출력"이나 "15초마다 방 인원 알림" 같은 주기 작업은 어떻게 만들까요? Timer를 따로 두지 않고, actor가 자기 자신에게 다음 일정을 예약합니다.

private void StatsHeartbeat()
{
    if (_stopped) return;

    Console.WriteLine($"==== Stats: 사용자 {_users.Count}명 ====");

    DoAsyncAfter(_statsPeriod, StatsHeartbeat);  // ← 자기 자신을 5초 뒤로 예약
}

ExampleChatServer는 이 패턴을 3군데에서 씁니다:

  • ChatServer — 5초마다 통계, 10초마다 유휴 스캔
  • Room — 15초마다 방 인원 알림

8유휴 disconnect 시퀀스 — actor가 actor에게 말하기

"30초 동안 아무 활동이 없으면 자동 종료"를 어떻게 구현할까요? 두 actor가 서로에게 메시지를 던지며 협력합니다.

왜 User가 직접 검사하나? _lastActivityTickMs는 메시지를 받을 때마다 갱신되는데, 만약 ChatServer가 그 값을 직접 읽으면 race가 생깁니다. User actor 안에서만 읽고 쓰면 락이 필요 없습니다 — 이게 actor 모델의 진짜 위력입니다.

9파일별 핵심 코드

ChatWorker.cs IRunnable

public class ChatWorker : IRunnable
{
    public static readonly ConcurrentQueue<Action> InboundCommands = new();

    public bool Run(CancellationToken token)
    {
        if (token.IsCancellationRequested) return false;

        if (InboundCommands.TryDequeue(out var cmd))
            cmd();   // ← 이 안에서 server.HandleX 가 실행됨
        else
            Thread.Sleep(1);

        long now = ThreadContext.TickCount;  // 워커 스레드 ThreadLocal
        // ... 5초마다 통계 로그 ...
        return true;
    }
}

전용 OS 스레드에서 무한 루프로 큐를 비웁니다. 일이 없으면 1ms 잠깐 양보.

User.cs AsyncExecutable

public sealed class User : AsyncExecutable
{
    private readonly HashSet<string> _joinedRoomIds = [];   // 락 없음!
    private long _lastActivityTickMs;

    public void DeliverMessage(ChatMessage msg) => DoAsync(() =>
    {
        _lastActivityTickMs = Environment.TickCount64;
        _client.SendMessage(msg);   // 동기 — User actor 큐에서 직렬
    });

    public UserSnapshot GetSnapshot()
    {
        using var ev = new ManualResetEventSlim(false);
        UserSnapshot? result = null;
        DoAsync(() =>
        {
            long idle = Environment.TickCount64 - _lastActivityTickMs;
            result = new UserSnapshot(UserId, Username, _joinedRoomIds.ToList(), idle);
            ev.Set();
        });
        ev.Wait();   // 차단 대기 — 전통적 스레드 동기화
        return result!;
    }

    public void CheckIdleAndDisconnect(ChatServer server, long thresholdMs) => DoAsync(() =>
    {
        long idle = Environment.TickCount64 - _lastActivityTickMs;
        if (idle > thresholdMs)
            server.HandleUserDisconnect(UserId);  // actor → actor
    });
}

Room.cs AsyncExecutable

public void ProcessChatMessage(string senderId, string content) => DoAsync(() =>
{
    if (!_users.TryGetValue(senderId, out var sender)) return;

    var message = new ChatMessage(...);

    foreach (var user in _users.Values)
        user.DeliverMessage(message);   // 각자 자기 큐로 넘기고 즉시 다음 사람
});

private void Heartbeat(TimeSpan period)
{
    if (_stopped) return;
    // ... 인원 알림 broadcast ...
    DoAsyncAfter(period, () => Heartbeat(period));   // 자기복제
}

ChatServer.cs AsyncExecutable

public void HandleRoomChat(string userId, string roomId, string content) => DoAsync(() =>
{
    if (_rooms.TryGetValue(roomId, out var room))
        room.ProcessChatMessage(userId, content);   // Room actor로 넘김
});

public ServerSnapshot GetSnapshot()
{
    // 1단계 — server actor 큐에서 user/room 참조만 복사
    User[] userArr; Room[] roomArr;
    using (var ev = new ManualResetEventSlim(false))
    {
        User[]? u = null; Room[]? r = null;
        DoAsync(() =>
        {
            u = _users.Values.ToArray();
            r = _rooms.Values.ToArray();
            ev.Set();
        });
        ev.Wait();
        userArr = u!; roomArr = r!;
    }
    // 2단계 — 호출자 스레드에서 fan-out (각 actor 가 자기 큐로 직렬화)
    var users = userArr.Select(x => x.GetSnapshot()).ToArray();
    var rooms = roomArr.Select(x => x.GetSnapshot()).ToArray();
    return new ServerSnapshot(users, rooms);
}

public void Stop()
{
    _stopped = true;
    DoAsync(() => { /* 모든 방 강제 정리 */ });
    DisposeAsync().AsTask().Wait();   // 라이브러리의 ValueTask 만 1회 차단
    _dispatcher?.Dispose();             // 워커 스레드 Join
    TimerRegistry.DisposeAll();
}

★유지보수: 람다 vs 명명된 메서드

actor 패턴의 흔한 함정 하나는 처리 본문을 모두 DoAsync(() => { ... }) 람다 안에 쑤셔넣는 것입니다. 디버거에서 스택 트레이스가 <HandleUserConnect>b__0_0 같은 익명 이름으로 찍히고, 브레이크포인트·grep·프로파일러 모두 곤란해집니다.

이 예제는 2단 분리 패턴을 사용합니다:

  • public Handle* / 동사* — 큐에 푸시만. 한 줄짜리 expression-bodied.
  • private Process* — 실제 본문. 디버거가 이름으로 잡고, 스택 트레이스에 또렷이 찍히고, grep으로 한 번에 찾을 수 있습니다.

❌ 람다에 본문 매몰

public void HandleUserConnect(IChatClient client)
{
    DoAsync(() =>
    {
        if (_users.ContainsKey(client.UserId))
        {
            Console.WriteLine($"이미 접속: {client.UserId}");
            return;
        }
        var user = new User(client);
        _users[user.UserId] = user;
        BroadcastSystemToAll(...);
    });
}

스택 트레이스: <HandleUserConnect>b__0_0()

✓ 본문은 명명된 메서드로

public void HandleUserConnect(IChatClient client)
    => DoAsync(() => ProcessUserConnect(client));

private void ProcessUserConnect(IChatClient client)
{
    if (_users.ContainsKey(client.UserId))
    {
        Console.WriteLine($"이미 접속: {client.UserId}");
        return;
    }
    var user = new User(client);
    _users[user.UserId] = user;
    BroadcastSystemToAll(...);
}

스택 트레이스: ChatServer.ProcessUserConnect()

왜 이게 더 좋은가?

  • 스택 트레이스 — 예외 발생 시 컴파일러가 만든 익명 이름(<...>b__#_#) 대신 사람이 읽을 수 있는 메서드 이름이 나옵니다.
  • 브레이크포인트 — 메서드 첫 줄에 한 번만 걸면 됨. 람다는 람다가 캡처된 위치마다 따로 걸어야 함.
  • 프로파일러 — VS Profiler / dotTrace / PerfView 모두 메서드 이름으로 시간 표시. 람다는 모두 묶여서 무엇이 느린지 안 보임.
  • 코드 검색 — grep ProcessUserConnect 한 번이면 정의·호출 모두 찾음.
  • 코드 리뷰 — 한 메서드 = 한 책임이 시각적으로 명확.
  • 테스트 — InternalsVisibleTo 또는 별도 인터페이스로 단위 테스트하기 쉬움.

인라인 람다를 그대로 둬도 되는 경우

  • 1-statement trivial 본문: DoAsync(() => _joinedRoomIds.Remove(roomId)) — 추출하면 오히려 노이즈.
  • 로컬 변수 closure 캡처가 필요한 경우: GetSnapshot()처럼 ManualResetEventSlim과 결과 변수를 캡처해야 하면 람다가 자연스럽습니다.
실용 규칙: 본문이 2줄 이상이거나 분기/반복이 있으면 Process* 메서드로 추출하세요. 그렇지 않으면 람다 유지.

★네트워크 IO 스레드 (멀티 프로듀서)

실제 채팅 서버는 여러 TCP 연결이 각자 자기 IO 스레드를 가지고 동시에 패킷을 파싱 → server.HandleX(...) 를 호출합니다. 이 예제의 NetworkSimulator 는 이 구조를 그대로 흉내 냅니다.

이 구조가 보여주는 4가지 핵심 특성

① 네트워크 스레드와 처리 스레드의 완전한 분리

IO 스레드는 패킷을 받아 큐에 던지면 즉시 다음 패킷으로 넘어갑니다. 워커는 자기 페이스로 actor 로직을 처리합니다. 두 그룹은 서로의 속도에 영향을 받지 않으며, 각자 다른 기준으로 스케일됩니다 (IO=연결 수, Worker=CPU 코어 수). Kestrel · libevent · boost.asio 등 모든 서버 프레임워크가 이 패턴입니다.

② 다중 Producer ConcurrentQueue 안전성 (lock-free)

전통적 큐는 push마다 lock을 잡아야 해서, N개 IO 스레드 중 1개만 들어갈 수 있고 나머지 N-1개는 대기합니다. ConcurrentQueue<T>는 CAS(Compare-And-Swap) 기반이라 N개 스레드가 모두 동시에 진행하며, 충돌 시 짧게 retry할 뿐 절대 block되지 않습니다.

③ 동시 push에도 Actor 직렬화는 유지됨

여러 IO 스레드가 같은 actor(예: ChatServer)에 동시에 명령을 던져도, actor의 본문은 여전히 한 번에 하나씩만 실행됩니다. _remainingTaskCount를 0→1로 increment 한 스레드만이 그 회차의 Flush를 잡고, 나머지는 채널에 쓰기만 하고 빠집니다. 이게 락 없이도 actor 내부 변수가 안전한 이유입니다.

④ 자동 부하 분산 (FIFO + 빈 워커가 가져감)

워커는 일이 없으면 큐를 polling 합니다. 일이 들어오면 가장 먼저 빈 워커가 가져갑니다. 사전 할당이나 라우팅 룰 없이도 자연스럽게 균등 분산됩니다. --network 6 4 실측에서 97건이 4 워커에 27/24/24/22 (편차 ±2) 로 분산되었습니다.

로그에서 무엇을 봐야 할까?

  • 워커 스레드 ID가 다 다름 (예: tid=8, 9, 10, 11) — 진짜 OS 스레드 4개가 떴다는 증거.
  • "[Worker N] 처리 X건" — 명령이 워커들에 분산되었다는 증거.
  • "==== [Stats] ====" 5초마다 출력 — 자기복제 heartbeat 동작.
  • "[Idle] alice 유휴 ..." — User가 자기 시각을 검사하고 disconnect를 요청한 흔적.

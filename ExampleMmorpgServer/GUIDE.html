SECTION 01

이 예제가 보여주려는 것

ExampleMmorpgServer는 진짜 게임을 굴리려는 게 아니라, JobDispatcherNET 라이브러리가 어떻게 동시성 문제를 푸는지를 MMO 서버라는 익숙한 시나리오로 보여주는 학습용 예제다. "여러 명의 플레이어가 같은 시각에 서로 공격하는데, 어떻게 데이터가 깨지지 않을까?" — 이 질문에 답한다.

🔓

락 없이 동시성

lock, Mutex, ReaderWriterLockSlim이 단 하나도 등장하지 않는다. 그런데도 데이터 경쟁이 일어나지 않는다.

🚦

같은 객체 = 자동 직렬

한 플레이어를 두 명이 동시에 공격해도, 그 플레이어의 HP 갱신은 한 번에 하나씩 안전하게 처리된다.

⚡

다른 객체 = 자동 병렬

서로 다른 플레이어의 이동·공격은 완전 병렬로 동시에 처리된다. 개발자가 신경 쓸 일이 없다.

⏱️

타이머도 락 없이

"3초마다 HP 회복", "5초 후 부활" 같은 주기·지연 작업도 DoAsyncAfter로 자기복제 한 줄.

📷

외부 read는 스냅샷

모니터링·디버깅을 위해 외부에서 상태를 읽을 때도 GetSnapshot 패턴으로 일관성 있게 받는다.

🛡️

예외도 한 곳에서

actor 큐 안에서 발생한 모든 예외는 AsyncExecutable.OnError 한 콜백으로 모인다.

🔀

IO ↔ 패킷 처리 분리

RecvIO/SendIO 전용 스레드가 동기 Read/Write를 맡고, actor Flush는 워커 스레드에서. IO 스레드는 actor 람다를 절대 실행하지 않는다.

🚫

async/await 0개

코드 전체에 async/await 키워드가 단 한 번도 등장하지 않는다. 대기는 전용 스레드에서, 종료는 1회 동기 차단으로.

한 줄 요약 · 이 라이브러리의 본질은 "객체마다 자기 큐를 주고, 큐는 한 번에 하나씩만 실행한다"는 단 하나의 규칙이다. 이 규칙으로부터 위 8가지 효과가 자연스럽게 따라온다.
SECTION 02

큰 그림 한눈에 보기

클라이언트가 쏜 패킷이 서버 안에서 어디로 흘러가는지 먼저 보자. 파란색은 큐(Queue), 보라색은 actor, 회색은 데이터 저장소다.

패킷이 들어오면 GameZone 큐 → 해당 PlayerActor 큐로 차례차례 흘러간다. 각 큐는 "한 번에 하나"라는 규칙으로 동작한다.

흐름을 한 문장으로

클라이언트가 패킷을 보내면 → RecvIO 스레드(세션 전용 OS 스레드)가 동기 Read로 받아 → 그 패킷을 세션 자체의 패킷 큐에 적재 + InboundCommands에 drain 명령을 1회 등록 → 워커 스레드가 그 명령을 꺼내 세션 큐를 순서대로 drain (한 세션은 한 워커가 순차로!) → PacketHandler → GameServer.HandleX로 → GameZone의 작업 큐에 "할 일"로 등록 → GameZone이 자기 차례에 그 일을 꺼내 처리하면서 → 해당 PlayerActor의 작업 큐에 다시 "할 일"로 등록 → PlayerActor가 자기 차례에 실제 로직(이동·데미지 적용 등)을 수행한다. → 응답이 필요하면 워커는 SendIO 스레드의 송신 큐에 push (워커는 절대 Write로 블로킹되지 않음).

두 가지 라우팅이 동시에 일어난다.
① 스레드 라우팅 — IO 스레드(Recv/Send) ↔ 워커 스레드 사이는 큐로만 연결. IO 스레드는 절대 actor 람다를 실행하지 않는다.
② actor 라우팅 — GameZone → PlayerActor의 두 단계 큐. 같은 플레이어를 동시에 공격해도 그 플레이어 큐 안에서 자연히 한 줄로 줄 서게 된다.
SECTION 03

핵심 개념 4가지

이 예제 코드를 이해하려면 JobDispatcherNET의 이 4가지만 알면 된다. 하나씩 그림과 함께 본다.

3.1 AsyncExecutable — "객체마다 자기 큐"

AsyncExecutable은 "자기만의 작업 큐를 가진 객체"의 부모 클래스다. 이 클래스를 상속받으면 그 객체는 다음 두 가지를 갖게 된다.

  • DoAsync(Action) — "이 일을 내 큐에 넣어두고 알아서 처리해"
  • DoAsyncAfter(TimeSpan, Action) — "n초 뒤에 내 큐에 넣어줘"

핵심 규칙: 한 객체의 큐는 한 번에 한 개의 작업만 실행한다. 그래서 객체 안의 필드를 락 없이 안전하게 읽고 쓸 수 있다.

// ExampleMmorpgServer/PlayerActor.cs (요약)
public class PlayerActor : AsyncExecutable
{
    private readonly Player _player;

    public void Move(float newX, float newY)
    {
        DoAsync(() =>
        {
            // 이 람다는 PlayerActor 큐 안에서 한 번에 하나만 실행됨.
            // → _player.X / _player.Y는 락 없이 안전하게 갱신할 수 있다.
            _player.X = newX;
            _player.Y = newY;
        });
    }
}

"같은 객체의 큐는 한 번에 하나"라는 규칙이 어떻게 보이는지 살펴보자.

시나리오 A · 같은 PlayerActor에 작업 3개 — 자동 직렬화
PlayerActor #1
Move
Damage
Move
AreaDamage
시간 →

하나가 끝나야 다음이 시작된다. _player.Hp 같은 필드를 읽고 쓰는 사이에 다른 작업이 끼어들 수 없다.

시나리오 B · 다른 PlayerActor 두 개 — 완전 병렬
PlayerActor A
Move
Damage
PlayerActor B
Move
Damage
Move

서로 다른 객체의 큐는 서로를 막지 않는다. 5명이 동시에 이동해도 5개의 PlayerActor가 같은 시각에 자기 일을 한다.

3.2 DoAsync — "메서드 직접 호출 ≠ DoAsync"

가장 흔한 초보 실수는 DoAsync가 그냥 비동기 호출이라고 생각하는 것이다. 실제로는 다르다.

❌ 직접 호출 
// 호출 즉시 실행됨
// 누가 호출했냐에 따라 어느 스레드
// 인지 알 수 없음 → 락 필요
player.X = newX;
player.Y = newY;
✅ DoAsync로 큐에 등록 
// 람다를 큐에 넣기만 함
// 실행은 actor 큐가 자기 차례에
// → 항상 직렬, 락 불필요
DoAsync(() => {
    player.X = newX;
    player.Y = newY;
});
코딩 컨벤션 · 이 예제는 public Handle*()는 큐에 푸시만 하고, 실제 본문은 private Process*()로 분리한다. 디버거에서 스택을 봤을 때 "어디서 락이 잠겼는지"가 아니라 "어떤 일이 큐에서 처리되고 있는지"가 한눈에 보인다. 
// GameZone.cs — 컨벤션 예시
public void HandleMove(string playerId, float nx, float ny)
    => DoAsync(() => ProcessMove(playerId, nx, ny));   // 큐에 푸시만

private void ProcessMove(string playerId, float nx, float ny)
{
    // 실제 본문 — GameZone 큐 안에서만 호출되므로 락 불필요
    if (_actors.TryGetValue(playerId, out var actor))
        actor.Move(nx, ny);
}

3.3 JobDispatcher와 IRunnable — 전용 OS 스레드

JobDispatcher<T>는 전용 OS 스레드 N개를 만들어 풀로 관리한다. 왜 .NET ThreadPool을 쓰지 않을까?

답은 ThreadLocal 안정성이다. 이 라이브러리는 각 워커 스레드마다 자기만의 TimerQueue(DoAsyncAfter가 쓰는 타이머)와 TickCount(현재 시각 캐시)를 두는데, 이게 안정적으로 살아있으려면 스레드가 사라지거나 다른 일에 끌려가면 안 된다. ThreadPool은 OS가 공유하는 풀이라 이 보장이 없다. 

// ExampleMmorpgServer/GameWorker.cs
public class GameWorker : IRunnable   // JobDispatcher가 사용할 워커 정의
{
    public bool Run(CancellationToken ct)
    {
        if (ct.IsCancellationRequested) return false;

        // ThreadContext.TickCount는 이 워커 전용 스레드의 ThreadLocal 시각.
        // 다른 워커와 완전히 독립 → 락 없이 워커별 주기 통계 가능.
        long now = ThreadContext.TickCount;
        if (now - _lastHeartbeatTick >= 5000)
        {
            _lastHeartbeatTick = now;
            Console.WriteLine($"[워커 #{_workerId}] alive tick={now}ms");
        }

        Thread.Sleep(1);     // 전용 스레드라 sleep해도 다른 사람 안 막음
        return true;          // true면 계속 실행, false면 종료
    }

    public void Dispose() { /* 종료 시 호출됨 */ }
}

실제 사용은 한 줄이다.

// GameServer.cs
_dispatcher = new JobDispatcher<GameWorker>(workerCount: 4);
_ = _dispatcher.RunWorkerThreadsAsync();    // 4개의 OS 스레드 시작

3.4 DoAsyncAfter — 자기복제 주기 작업

"3초마다 HP 회복" 같은 주기 작업을 흔히 Timer나 Task.Delay로 만들지만, 이 라이브러리에서는 더 단순한 패턴이 있다. "자기 자신을 다음 타이밍에 자기 큐에 다시 등록한다." 

// PlayerActor.cs — 자가 회복 heartbeat
public void StartRegen() => DoAsync(RegenTick);

private void RegenTick()
{
    if (_despawned) return;

    if (_player.IsAlive && _player.Hp < _player.MaxHp)
    {
        _player.Hp = Math.Min(_player.MaxHp, _player.Hp + 30);
        Console.WriteLine($"  [{_player.Name}] 자동회복");
    }

    // 자기복제 — 3초 뒤에 자기 큐에 다시 RegenTick을 등록
    DoAsyncAfter(TimeSpan.FromSeconds(3), RegenTick);
}
애니메이션 · RegenTick의 자기복제 사이클
큐
① 큐 대기
② 실행 (Hp+=30)
③ 3초 타이머
RegenTick

람다가 큐 → 실행 → 타이머 → 다시 큐로 무한 루프. 외부 스레드/타이머 객체가 없으므로 락이 0개다.

같은 패턴으로 GameZone의 5초마다 인원 통계도 만들어진다.

// GameZone.cs
private void Heartbeat(TimeSpan period)
{
    if (_stopped) return;
    if (_actors.Count > 0) {
        Console.WriteLine($"  [{_name}] heartbeat 인원={_actors.Count}");
    }
    DoAsyncAfter(period, () => Heartbeat(period));    // 자기복제
}

그리고 1회성 지연 작업도 같은 함수다.

// PlayerActor.cs — 사망 후 5초 뒤 부활 (1회만)
DoAsyncAfter(TimeSpan.FromSeconds(5), () => {
    _player.Hp = _player.MaxHp;
    // 부활 처리...
});

4.1 Player — "그냥 데이터"

Player는 actor가 아니다. AsyncExecutable을 상속받지 않는다. 그저 이름·좌표·HP 같은 상태 필드를 가진 평범한 데이터 객체다. 안전성은 "이 객체를 누가 만지느냐"로 보장한다 — 항상 그 플레이어의 PlayerActor 큐 안에서만 만진다. 

// Player.cs
public class Player
{
    public string PlayerId { get; }
    public float X { get; set; }    // 락 없음 — PlayerActor 큐 안에서만 set
    public float Y { get; set; }
    public int   Hp { get; set; }
    public Action<string>? SendPacket { get; set; }   // 송신 콜백

    public int TakeDamage(int raw)
    {
        int dmg = Math.Max(1, raw - Defense);
        Hp = Math.Max(0, Hp - dmg);
        return dmg;
    }
}

4.2 PlayerActor — 플레이어 한 명의 두뇌

각 플레이어마다 한 개씩 만들어진다. AsyncExecutable을 상속해서 자기만의 작업 큐를 가진다. 이 큐 덕분에 같은 플레이어에 대한 작업은 자동으로 순서가 보장되고, 다른 플레이어와는 병렬로 처리된다.

주요 메서드 분류:

입력 처리

Move, MeleeAttack, AreaAttack — 자신이 발생시키는 행동.

입력 수신

ReceiveMeleeDamage, ReceiveAreaDamage — 다른 actor가 주는 데미지.

주기 작업

StartRegen → RegenTick — DoAsyncAfter 자기복제 회복.

라이프사이클

Despawn, HandleDeath — 종료 처리, 5초 뒤 부활.

한 가지 패턴이 반복된다 — actor → actor 메시지 전달은 항상 "스냅샷"으로.

// PlayerActor.cs — 근접 공격
public void MeleeAttack(string targetId, float meleeRange)
{
    DoAsync(() =>
    {
        if (_despawned || !_player.IsAlive) return;

        // 1단계: 공격자 정보를 불변 record struct로 "복사"한다.
        // 이렇게 안 하고 _player를 그대로 넘기면 대상 actor 큐에서 _player.X를
        // 읽는 사이에 내가 _player.X를 갱신해 race가 발생할 수 있다.
        var snapshot = new AttackerSnapshot(
            _player.PlayerId, _player.Name,
            _player.X, _player.Y, _player.Attack);

        // 2단계: 대상 lookup은 GameZone 큐로 위임 → race 없음
        _zone.SendMeleeDamage(targetId, snapshot, meleeRange);
    });
}
왜 record struct인가? · AttackerSnapshot은 readonly record struct — 불변 + 값 타입이다. 한 번 만들고 나면 바뀔 수 없으므로 어떤 스레드가 읽어도 항상 같은 값을 본다. 스레드 간 데이터 전달은 항상 불변으로 — actor 모델의 황금률이다.

4.3 GameZone — 월드의 우체국

GameZone도 AsyncExecutable이다. 자기만의 큐가 있고, "어떤 패킷이 어떤 PlayerActor로 가야 하는지"를 결정하는 라우터 역할이다. 플레이어 사전(_actors)을 들고 있는데, 일반 Dictionary로 충분하다 — 모든 접근이 GameZone 큐 안에서 일어나므로. 

// GameZone.cs
public class GameZone : AsyncExecutable
{
    private readonly Dictionary<string, PlayerActor> _actors = [];   // 락 없음!
    private readonly SpatialIndex _spatialIndex;

    // 외부 진입점은 모두 큐에 푸시만 (코딩 컨벤션)
    public void EnterZone(Player p, float sx, float sy)
        => DoAsync(() => ProcessEnterZone(p, sx, sy));

    public void HandleMove(string id, float nx, float ny)
        => DoAsync(() => ProcessMove(id, nx, ny));

    // PlayerActor → GameZone으로 위임되는 melee 라우팅
    internal void SendMeleeDamage(string targetId, AttackerSnapshot atk, float range)
        => DoAsync(() =>
        {
            if (_actors.TryGetValue(targetId, out var target))
                target.ReceiveMeleeDamage(atk, range);
        });
}
주의 · 만약 _actors를 외부 스레드(예: 네트워크 IO 스레드)에서 직접 만지면 race가 생긴다. 그래서 GameZone도 actor가 되어야 한다 — 모든 접근을 자기 큐 안으로 모은다.

4.4 GameServer — 조립공장

GameServer는 actor가 아니다. 그냥 JobDispatcher와 GameZone의 수명주기를 묶어주는 조립공장이다. 외부 진입점도 모두 GameZone으로 단순 위임한다.

Start/Stop 모두 동기다. 라이브러리가 반환하는 ValueTask는 종료 경로에서 .AsTask().Wait() 한 번만 차단해 처리한다 — 코드 전체에 async/await가 한 번도 등장하지 않는다. 

public void Start()
{
    _dispatcher = new JobDispatcher<GameWorker>(_workerCount);
    _ = _dispatcher.RunWorkerThreadsAsync();           // 워커 풀 시작 (Task 즉시 반환)
    _zone.StartHeartbeat(TimeSpan.FromSeconds(5));   // 자기복제 통계
}

public void Stop()
{
    // ValueTask를 1회만 차단해 동기적으로 drain
    _zone.DisposeAsync().AsTask().Wait();   // actor 큐들 drain
    _dispatcher?.Dispose();                  // 워커 스레드 join (동기)
    TimerRegistry.DisposeAll();             // 비-워커 TimerQueue 정리
}
왜 async/await을 안 쓰나? · 실제 게임 서버는 IO 대기를 전용 OS 스레드로 처리하므로 async 상태머신 비용을 지불할 이유가 없다. 패킷 처리 자체에는 대기가 없고, 대기가 발생하는 지점(소켓 Read/Write, actor 큐 drain 종료)은 동기 차단으로 짧게 끝난다. 자세한 설명은 Section 8 참고.

4.5 NetworkServer — IO 스레드와 패킷 처리 스레드의 완전 분리

네트워크 모듈의 설계 원칙은 단순하다. "IO 스레드는 절대로 actor의 람다를 직접 실행하지 않는다." Read/Write 같은 IO 대기는 전용 OS 스레드에서, 게임 로직(actor Flush)은 JobDispatcher 워커 스레드에서. 두 영역이 큐 하나로만 연결된다.

스레드 구성 — 한 세션에 IO 스레드가 두 개

📥

RecvIO 스레드

동기 Read로 블로킹. 줄 단위로 잘라 세션 자체의 패킷 큐에 push 한 뒤 InboundCommands에 drain 명령을 1회 등록하고 즉시 다음 Read로 돌아간다.

📤

SendIO 스레드

세션의 BlockingCollection에서 byte[]를 꺼내 동기 Write. 워커가 SendPacket을 호출하면 큐에 push만 — 송신 대기로 워커가 막히지 않는다.

👂

Accept 스레드

전용 한 개. AcceptTcpClient()를 동기 호출하며 새 연결을 받아 위 두 스레드를 띄운다.

⚙️

Worker 스레드 (JobDispatcher)

InboundCommands에서 명령을 dequeue 하여 세션의 패킷 큐를 drain → GameServer.HandleX → actor Flush. 실제 게임 로직은 모두 여기서 실행된다.

RecvIO — 받기만 하고 즉시 빠져나간다

// NetworkServer.cs — RecvLoop 요약 (동기 Read)
private void RecvLoop()
{
    var buffer = new byte[1024];
    var pendingText = new StringBuilder();   // 토막난 패킷 보관
    while (true)
    {
        int n = _stream.Read(buffer, 0, buffer.Length);   // 동기 블로킹
        if (n == 0) break;

        pendingText.Append(Encoding.UTF8.GetString(buffer, 0, n));

        bool pushedAny = false;
        while (/* '\n' 발견 */) {
            _packetQueue.Enqueue(line);   // 세션 자체의 큐로 적재
            pushedAny = true;
        }

        if (pushedAny)
            TryScheduleDrain();           // 워커 큐에 drain 명령 1회 등록
    }
}
핵심 · RecvIO 스레드는 PacketHandler.Handle이나 GameServer.HandleX를 직접 호출하지 않는다. 직접 호출하면 actor 큐가 비어있을 때 AsyncExecutable.DoTask의 leader 경로가 IO 스레드에서 Flush를 실행해 버린다. 이 한 줄의 우회가 "IO 스레드와 패킷 처리 스레드 완전 분리"의 핵심.

세션별 drain — 같은 클라이언트의 패킷 순서 보장

여러 워커가 InboundCommands에서 동시에 명령을 꺼내 _zone.DoAsync를 부르면, 라이브러리의 Increment + TryWrite가 원자적이지 않아서 같은 세션의 EnterZone → Move가 채널 안에서 Move → EnterZone으로 뒤집힐 수 있다 (실제 발견된 race). 이를 막기 위해 한 세션의 패킷은 항상 한 워커가 순차로 drain 한다. 

// 세션별 큐 + CAS 락 = "한 시점에 한 워커만 이 세션을 drain"
private readonly ConcurrentQueue<string> _packetQueue = new();
private int _drainScheduled;   // 0=idle, 1=drain 중/예약

private void TryScheduleDrain()
{
    // CAS로 한 스레드만 drain 명령을 InboundCommands에 등록한다.
    if (Interlocked.CompareExchange(ref _drainScheduled, 1, 0) != 0)
        return;
    GameWorker.InboundCommands.Enqueue(DrainPackets);
}

private void DrainPackets()    // 워커 스레드에서 실행
{
    try
    {
        while (_packetQueue.TryDequeue(out var line))
        {
            if (ReferenceEquals(line, DisconnectMarker))
                _gameServer.HandlePlayerDisconnect(Player.PlayerId);
            else
                PacketHandler.Handle(_gameServer, Player, line);
        }
    }
    finally
    {
        Volatile.Write(ref _drainScheduled, 0);
        // drain 종료와 RecvIO push 사이 race 처리
        if (!_packetQueue.IsEmpty)
            TryScheduleDrain();
    }
}
Disconnect도 같은 큐로 · 연결 종료 처리 역시 DisconnectMarker라는 sentinel 문자열로 같은 _packetQueue에 넣는다. 그래야 큐에 줄 서 있던 EnterZone/Move/Attack이 모두 처리된 뒤에야 HandlePlayerDisconnect가 실행된다. ReferenceEquals로 식별하므로 사용자가 같은 문자열을 보내도 절대 일치하지 않는다 (서로 다른 string 인스턴스).

SendIO — 송신 대기는 워커가 아니라 전용 스레드에서

// 워커가 actor 람다 안에서 호출하는 것은 큐 push 하나뿐
private void EnqueueSend(string message)
{
    try
    {
        _sendQueue.Add(Encoding.UTF8.GetBytes(message + "\n"));
    }
    catch (InvalidOperationException) { /* CompleteAdding/Disposed — 무시 */ }
}

// 전용 SendIO 스레드 — 동기 Write가 블로킹되어도 워커는 영향 없음
private void SendLoop()
{
    foreach (var data in _sendQueue.GetConsumingEnumerable())
    {
        try { _stream.Write(data, 0, data.Length); }
        catch { break; }
    }
    CloseSocket();   // 송신 실패 시 RecvLoop도 깨운다
}
왜 BlockingCollection을 Dispose하지 않나? · 워커가 PlayerActor 큐에 남은 잔여 패킷을 늦게 처리하면서 SendPacket을 호출하는 race가 흔하다. 이때 큐가 disposed라면 ObjectDisposedException이 actor Flush 안에서 터진다. CompleteAdding만 호출하고 Dispose는 GC에 맡기는 편이 안전하다.

4.6 SpatialIndex — 유일하게 락이 있는 곳

AoE 공격 같은 "주변 5m 안에 있는 모든 플레이어 찾기"를 빠르게 하기 위한 격자 기반 공간 자료구조. 이건 ConcurrentDictionary를 쓴다 — 여러 PlayerActor가 자기 위치를 갱신하면서 동시에 GameZone이 범위 조회를 하기 때문.

이 라이브러리 전체에서 "락"이 등장하는 유일한 곳이 바로 ConcurrentDictionary의 내부 버킷 락뿐이다. 외부 락은 0개.

SECTION 05

코드 호출 흐름

실제 시나리오에서 코드가 어떤 순서로 호출되는지 시퀀스 다이어그램으로 본다. 가로축은 actor, 세로축은 시간이다.

5.1 플레이어 입장 (EnterZone)

패킷 EnterZone|field|50|50이 들어오면:

RecvIO 스레드는 ①②만 하고 즉시 다음 Read로. 실제 작업(actor 생성, spatial 등록, regen 시작)은 워커가 시작하고 actor 큐들이 이어받아 직렬 처리.

5.2 근접 공격 (A → B)

이 흐름이 actor 모델의 진수다. 두 actor 사이의 메시지 패싱이 4번의 큐 hop으로 일어난다.

A의 상태 읽기는 A 큐에서, B의 상태 변경은 B 큐에서. 둘 사이에 흐르는 건 불변 스냅샷 한 덩어리뿐.

흐름의 핵심을 코드로 다시 보면:

// ① 외부에서
gameServer.HandleMeleeAttack("A", "B");
// ↓ GameZone.HandleMeleeAttack → DoAsync(ProcessMeleeAttack)
// ↓ ProcessMeleeAttack: _actors.TryGetValue → attacker.MeleeAttack("B", range)

// ③ A 큐 안에서
DoAsync(() => {
    var snap = new AttackerSnapshot(...);
    _zone.SendMeleeDamage("B", snap, range);
    // ↓ GameZone.DoAsync(_actors[B].ReceiveMeleeDamage(snap, range))
});

// ⑤ B 큐 안에서 (다른 스레드에서 병렬로 실행 가능)
DoAsync(() => {
    if (_player.DistanceTo(snap.X, snap.Y) > range) return;
    int dmg = _player.TakeDamage(snap.Attack);
    _player.SendPacket?.Invoke($"DAMAGE|...");
});

5.3 범위 공격 (AoE) — fan-out

범위 공격은 한 발이 여러 명에게 가는 흐름이다. SpatialIndex 조회를 GameZone 큐 안에서 하고, 각 대상에게 따로 DoAsync로 데미지를 보낸다 — 대상별 병렬 처리! 

// PlayerActor.AreaAttack — 공격자 큐 안에서
DoAsync(() =>
{
    var snap = new AttackerSnapshot(...);
    _zone.DispatchAreaDamage(snap, cx, cy, radius);   // GameZone에 위임
});

// GameZone.DispatchAreaDamage — GameZone 큐 안에서
DoAsync(() =>
{
    var nearby = _spatialIndex.QueryRadius(cx, cy, radius);
    foreach (var p in nearby)
    {
        if (p.PlayerId == snap.PlayerId) continue;
        if (_actors.TryGetValue(p.PlayerId, out var ta))
            ta.ReceiveAreaDamage(snap, cx, cy, radius);   // 각자의 큐로 분배 → 병렬
    }
});

대상 4명의 데미지 처리가 같은 시각에 동시에 일어난다. 락이 0개. 각 대상의 HP 갱신은 각자의 큐 안에서 직렬.

5.4 사망 → 5초 후 부활 (DoAsyncAfter 1회)

HP가 0이 되면 HandleDeath가 호출되고, 5초 뒤 부활 람다를 자기 큐에 예약한다.

// PlayerActor.HandleDeath — B 큐 안에서
private void HandleDeath(string killerName)
{
    Console.WriteLine($"  ★ {_player.Name} 사망!");
    _player.SendPacket?.Invoke($"DEATH|{killerName}");

    // 5초 뒤 자기 큐에 부활 작업 예약 (외부 타이머 객체 없음)
    DoAsyncAfter(TimeSpan.FromSeconds(5), () =>
    {
        if (_despawned) return;            // 그 사이에 디스폰됐으면 무시

        _player.Hp = _player.MaxHp;
        _player.X = ...; _player.Y = ...;
        _zone.Spatial.UpdatePosition(_player, oldX, oldY);
        _player.SendPacket?.Invoke($"RESPAWN|...");
    });
}
죽은 채로 들어오는 데미지는 어떻게? · HP가 0이어도 큐에는 다른 공격자들의 ReceiveMeleeDamage가 줄 서 있을 수 있다. 람다 첫 줄의 if (!_player.IsAlive) return; 체크 하나로 안전하게 흘려보낸다. 출력 로그에 "마법사이영희 이미 사망 — 전사김철수의 공격 무효"가 그것.

5.5 자동 회복 (자기복제 RegenTick)

입장하면 StartRegen()이 호출되고, 그 다음부터는 영원히 자기 자신을 다시 큐에 등록한다.

// 입장 시 한 번만 시동
actor.StartRegen();   // → DoAsync(RegenTick)

private void RegenTick()
{
    if (_despawned) return;            // 디스폰되면 사슬이 끊긴다 → 자연 종료

    if (_player.IsAlive && _player.Hp < _player.MaxHp)
        _player.Hp = Math.Min(_player.MaxHp, _player.Hp + 30);

    DoAsyncAfter(TimeSpan.FromSeconds(3), RegenTick);   // 다시 자기 자신
}
핵심 · 외부의 System.Timers.Timer 같은 객체가 필요 없다. DoAsyncAfter가 알아서 actor의 TimerQueue를 통해 정해진 시간 뒤에 자기 큐로 람다를 다시 보내준다. 락 0개. 종료도 단순 — _despawned 체크 한 줄로 사슬이 끊긴다.
SECTION 06

왜 이렇게 만들었나 — 설계 Q&A

코드를 읽다 보면 떠오르는 자연스러운 질문들에 답한다. 각 답이 "이 코드의 왜"를 풀어 준다.

왜 GameZone도 actor로 만들었나? ConcurrentDictionary면 충분하지 않나?
얼핏 충분해 보이지만 _actors의 한 항목과 그 항목이 가리키는 PlayerActor·SpatialIndex의 변경을 묶어서 일관성 있게 처리할 곳이 필요하다. 예를 들어 LeaveZone은 _actors에서 제거 + spatial 제거 + 잔여 작업 흐름 정리를 함께 해야 하는데, 이걸 외부 IO 스레드에서 직접 하면 동시에 들어온 Move와 race가 난다. GameZone을 actor로 만들면 이 모든 묶음이 자동으로 직렬화되어 race가 없어진다.
그럼 _actors가 일반 Dictionary여도 되는 이유는?
모든 접근이 GameZone의 큐 안에서만 일어나기 때문이다. "한 번에 하나만 실행" 규칙이 곧 단일 스레드 접근을 보장하므로 ConcurrentDictionary가 줄 락이 불필요하다. 이게 actor 모델의 핵심 가성비 — 락을 안 쓰는데도 안전한 이유.
SpatialIndex는 왜 ConcurrentDictionary인가? 일관성 위반 아닌가?
SpatialIndex는 여러 actor에 걸쳐 공유된다. 각 PlayerActor가 자기 위치를 갱신하면서, 동시에 GameZone이 AoE 범위 조회를 한다. 각 작업이 actor의 큐 밖에서도 발생하기 때문에 이 자료구조 자체가 스레드 안전이어야 한다. 반면 _actors는 GameZone 큐 안에서만 만져지므로 일반 Dictionary로 충분.
Player.X / Y / Hp는 왜 평범한 필드인가? volatile도 lock도 없다.
그 필드들은 오직 그 플레이어의 PlayerActor 큐 안에서만 읽고 쓰여진다. "한 큐는 한 번에 하나만 실행" 규칙이 단일 스레드 접근을 보장하므로, 단순 필드만으로 안전하다. "스냅샷"으로 다른 actor에 전달할 때만 record struct로 값을 복사한다 — 그래서 AttackerSnapshot이 등장한다.
_despawned는 왜 volatile인가?
_despawned 플래그를 쓰는 곳은 actor 큐 밖에서도 읽힐 수 있다. 예를 들어 종료 직전에 GameZone이 actor 컬렉션을 순회하면서 잔여 상태를 확인할 때. 이런 cross-thread 단일-bool read는 가벼운 메모리 가시성 보장이 필요해서 volatile로 둔다. 단일 bool에 대한 race는 의미가 없고(true/false 둘 중 하나), volatile은 그 쓰기/읽기가 다른 스레드에 즉시 보이도록만 보장한다.
Despawn은 왜 GameZone에서 직접 spatial을 정리하지 않고 PlayerActor 큐로 보내는가?
SpatialIndex.Remove가 일어나는 시점에 그 PlayerActor 큐에는 아직 처리되지 않은 Move나 Damage가 줄 서 있을 수 있다. 만약 GameZone이 Spatial.Remove를 먼저 호출해 버리면, 잔여 Move가 spatial에 위치를 다시 등록할 위험이 있다. "같은 actor의 작업은 같은 큐에서 순서대로"라는 규칙으로 정렬하기 위해, Despawn도 actor의 큐 마지막에 넣는다.
GetSnapshot은 왜 ManualResetEventSlim으로 동기 대기하는가? Task로 안 되나?
Task로도 가능하지만 actor 큐 안에서 결과를 한 번 캡처하면 끝이라 ManualResetEventSlim이 더 가볍다. 중요한 제약: 호출자는 다른 actor의 Flush 컨텍스트가 아니어야 한다. 만약 actor의 람다 안에서 GetSnapshot을 부르면 자기가 자기를 기다리는 데드락이 가능하다. 그래서 디버깅·모니터링 같은 외부(메인 스레드, REPL) 호출에서만 쓴다.
JobDispatcher 워커는 actor 작업을 어떻게 처리하나? 외부에서 어떻게 워커로 일을 넘기지?
이 예제(MMORPG)에서는 워커가 능동적으로 일감을 가져온다. 외부(IO 스레드, 메인 스레드)는 GameWorker.InboundCommands라는 ConcurrentQueue<Action>에 명령을 push만 한다. 워커의 Run() 메서드가 한 틱마다 그 큐에서 dequeue 하여 invoke 하고, 그 invoke 가 결국 GameServer.HandleX → DoAsync → Flush를 호출하므로 actor의 Flush가 워커 스레드에서 일어난다. 라이브러리 자체는 "DoAsync를 처음 호출한 스레드가 Flush 한다"는 단순한 규칙뿐이다 — 누가 호출자가 되느냐를 외부 큐로 통제해 IO 스레드와 패킷 처리 스레드를 분리한 것이 이 예제의 설계 포인트.
시뮬레이션 모드도 같은 InboundCommands를 쓰나?
그렇다. 시뮬레이션의 메인 스레드도 네트워크 모드의 IO 스레드와 똑같이 GameWorker.InboundCommands.Enqueue(() => gameServer.HandleX(...)) 패턴으로 명령만 등록한다. 실제 처리는 모두 워커 스레드에서 일어나므로 두 모드의 코드 흐름이 동일하다 — 콘솔 로그의 [스레드:N] 표시가 워커 스레드 ID 또는 timer 스레드 ID로 찍히는 것이 그 증거.
왜 async/await을 안 쓰나? .NET이라면 그게 표준 아닌가?
이 서버에서 진짜 IO 대기는 전용 OS 스레드가 맡는다 — RecvIO 스레드의 동기 Read, SendIO 스레드의 동기 Write, Accept 스레드의 동기 AcceptTcpClient. 패킷 처리 자체에는 어떤 대기도 없다(actor 큐 push는 즉시 반환, in-memory 작업뿐). 그러므로 async 상태머신이 만드는 컨티뉴에이션·스레드 풀 점유·Task allocation 같은 비용을 지불할 이유가 없다. 단 한 곳에서 동기 차단이 필요한데, 종료 경로의 _zone.DisposeAsync().AsTask().Wait()다. 라이브러리가 반환하는 ValueTask를 단 한 번만 차단해 흡수한다.
"IO 스레드와 패킷 처리 스레드 완전 분리" — 정확히 어떻게 보장하나?
AsyncExecutable.DoTask는 leader 경로에서 호출 스레드가 직접 Flush를 수행한다. 만약 IO 스레드가 직접 server.HandleX를 호출하면, GameZone 큐가 비어있을 때 IO 스레드가 leader가 되어 actor 람다를 실행해 버린다. 이 예제는 그래서 IO 스레드가 절대 게임 코드를 직접 호출하지 않는다. 대신 GameWorker.InboundCommands에 Action을 push만 하고 즉시 다음 Read로. Action을 dequeue해 invoke 하는 순간이 워커 스레드이므로, leader는 항상 워커가 된다.
같은 클라이언트의 패킷 순서는 어떻게 보장되나? 워커가 4개라면 race가 나지 않나?
실제로 발견된 race가 있었다. 4개 워커가 동시에 InboundCommands에서 한 클라이언트의 4개 패킷(EnterZone, Move, AreaAttack, Disconnect)을 dequeue한 뒤 각자 _zone.DoAsync를 호출하면, 라이브러리 내부의 Increment + TryWrite가 원자적이지 않아서 채널 안에서 순서가 뒤집힐 수 있다 — Move가 EnterZone보다 먼저 실행되어 actor 미등록 상태로 무시되는 식. 해결책: 한 세션의 패킷은 항상 같은 한 워커가 순차로 drain. 세션마다 자기 ConcurrentQueue<string>를 두고, CAS 락 _drainScheduled로 한 시점에 한 워커만 그 세션의 큐를 drain 한다 — RecvIO는 push만, drain 명령은 InboundCommands에 1회만 등록. 자세한 패턴은 Section 8.
송신(SendPacket)은 왜 별도 스레드가 필요한가? 워커가 그냥 Write 하면 안 되나?
NetworkStream.Write는 동기 호출이고, TCP 송신 버퍼가 가득 차면 블로킹된다. actor의 람다는 워커 스레드에서 실행되므로, 거기서 Write를 직접 부르면 슬로우 클라이언트 한 명이 워커를 통째로 점유한다 — 다른 모든 actor의 작업이 멈춘다. 그래서 세션마다 BlockingCollection<byte[]> 송신 큐와 전용 SendIO 스레드를 둔다. 워커는 큐에 push만 하고 즉시 빠져나오고, 실제 Write 블로킹은 SendIO 스레드가 먹는다 — 이 스레드가 막혀도 게임 로직은 그대로 진행된다.
AsyncExecutable.OnError는 한 번만 설정하는데 어떻게 모든 actor에 적용되나?
OnError는 정적(static) 콜백이다. 라이브러리 내부의 Flush가 람다를 실행할 때 try/catch로 감싸고, 잡힌 예외를 이 콜백으로 보낸다. 덕분에 코드의 어떤 actor에서 예외가 나도 한 곳에서 받을 수 있다. 이 예제에서는 Program.cs 첫 줄에서 단 한 번 설정한다.

SECTION 08

동기 IO와 스레드 분리 — 한 자리에서

이 예제가 일반적인 .NET 서버 샘플과 가장 다른 점이 이 부분이다. async/await 미사용 + IO 스레드와 패킷 처리 스레드의 완전 분리를 어떻게 보장하고 그 과정에서 어떤 race를 만났고 어떻게 풀었는지 한 곳에 모았다.

8.1 왜 async/await 없이 가는가

async/await의 가치는 "IO 대기 중에 스레드를 양보해 더 많은 동시 요청을 처리"하는 것이다. 이 가치는 스레드가 비싸고 IO가 자주 막히는 환경에서 빛난다 — 예를 들어 DB·HTTP API 호출이 많은 백엔드.

실시간 게임 서버는 다르다.

  • IO 대기는 거의 발생하지 않는다. 패킷이 도착하면 즉시 처리하고 다음을 기다린다. 처리 자체는 in-memory actor 작업이라 대기가 0.
  • 대기가 발생하는 지점은 따로 있다. 소켓 Read/Write의 블로킹, accept 대기 — 이건 전용 OS 스레드가 먹는다.
  • 전용 스레드는 비싸지 않다. 동시 접속 N명에 대해 (Recv + Send) × N + 워커 K 정도. 진짜 대규모(만 단위)에서는 IOCP/epoll로 가지만, 그것도 async가 아니라 다른 모델이다.

그러므로 이 예제는 async/await을 쓰지 않는다. 대신 "대기는 전용 스레드에서, 처리는 워커 스레드에서"로 명확하게 역할을 나눈다.

딱 하나의 동기 차단 · 종료 경로의 _zone.DisposeAsync().AsTask().Wait()가 유일한 sync-over-async 호출이다. 라이브러리가 노출하는 ValueTask를 1회 흡수한다. 그 외에는 코드 전체에 async/await가 한 번도 없다.

8.2 스레드 지도 — 누가 어떤 일을 하나

스레드 개수 맡는 일
NetAccept 1 동기 AcceptTcpClient()로 새 TCP 연결을 받음. 연결마다 RecvIO + SendIO 스레드 한 쌍을 띄운다.
NetRecv-{playerId} 세션당 1 동기 Read로 블로킹. 패킷을 줄 단위로 잘라 세션 자체의 _packetQueue에 push 후 InboundCommands에 drain 명령 1회 등록.
NetSend-{playerId} 세션당 1 BlockingCollection에서 byte[]를 꺼내 동기 Write. Write 블로킹은 여기서만 일어나므로 워커는 절대 막히지 않는다.
JobWorker-{N} 설정값 (기본 4) InboundCommands에서 명령을 dequeue해 invoke. 그 invoke 가 HandleX → DoAsync → Flush를 트리거하므로 actor Flush가 워커에서 실행됨.
Timer 스레드 전용 스레드별 1 (lazy) DoAsyncAfter로 예약된 작업을 정해진 시점에 actor 큐에 다시 푸시. 라이브러리의 TimerQueue가 ThreadLocal로 만든다.
핵심 규칙 · RecvIO/SendIO 스레드는 절대로 HandleX·DoAsync·Flush를 직접 호출하지 않는다. 그렇게 호출하면 AsyncExecutable의 leader 경로가 IO 스레드 위에서 actor 람다를 실행해 버린다 — "IO와 처리 분리"가 깨진다.

8.3 패킷이 큐를 거치는 두 단계

한 패킷이 IO 스레드에서 actor까지 가는 길에 큐를 두 번 거친다.

①은 세션별 큐(per-session) — 같은 클라이언트 패킷의 순서를 보장한다. ②는 전역 큐(global) — 어떤 워커가 이 세션을 다음에 처리할지 선택한다. 두 큐의 역할이 다르다.

8.4 발견된 race — Move가 EnterZone보다 먼저 실행된 사건

초기 구현은 RecvIO가 패킷마다 InboundCommands에 한 개의 Action을 push 했다. 그러면 4개 워커가 동시에 그 Action들을 dequeue하여 각자 _zone.DoAsync를 호출하는데... 

// AsyncExecutable.DoTask 의 leader 경로 (간략화)
if (Interlocked.Increment(ref _remainingTaskCount) > 1) {
    _jobQueue.Writer.TryWrite(task);   // follower
} else {
    _jobQueue.Writer.TryWrite(task);   // leader — Increment 와 별도 단계
    ...
}

문제는 Increment와 TryWrite가 원자적이지 않다는 점이다. 시간순으로 다음과 같은 인터리브가 가능하다.

  1. 워커 A: Increment → 1 (leader 경로 진입)
  2. 워커 B: Increment → 2 (follower 경로). TryWrite(Move) — 채널에 Move가 먼저 들어감
  3. 워커 A: TryWrite(EnterZone) — Move 다음에 들어감
  4. 워커 A: Flush 시작 → 채널에서 Move를 먼저 꺼내 ProcessMove 실행 → _actors에 actor 미등록 → 무시
  5. 이어서 EnterZone 실행 — 늦었음

실제 TCP 테스트로 이 reorder를 재현했다. 클라이언트가 EnterZone | Move | AreaAttack 세 패킷을 한 번에 보내면 Move 로그가 사라졌다.

해결 — 세션별 단일 워커 drain

한 세션의 패킷은 항상 같은 한 워커가 순차로 처리하면 race가 사라진다. 한 워커가 단일 producer로 _zone.DoAsync를 부르므로 채널 안에서도 자연히 순서 유지. 

// ClientSession.cs — 핵심 4가지 필드
private readonly ConcurrentQueue<string> _packetQueue = new();
private int _drainScheduled;             // 0=idle, 1=drain 중
private const string DisconnectMarker = "\0__DISCONNECT__\0";

private void TryScheduleDrain()
{
    // CAS 로 권한을 획득한 한 스레드만 InboundCommands 에 명령 등록
    if (Interlocked.CompareExchange(ref _drainScheduled, 1, 0) != 0) return;
    GameWorker.InboundCommands.Enqueue(DrainPackets);
}

private void DrainPackets()
{
    try
    {
        while (_packetQueue.TryDequeue(out var line))
        {
            if (ReferenceEquals(line, DisconnectMarker))
                _gameServer.HandlePlayerDisconnect(Player.PlayerId);
            else
                PacketHandler.Handle(_gameServer, Player, line);
        }
    }
    finally
    {
        Volatile.Write(ref _drainScheduled, 0);
        // drain 종료 ↔ RecvIO push 사이 race 처리
        if (!_packetQueue.IsEmpty) TryScheduleDrain();
    }
}
왜 잘 동작하는가? · 한 세션의 _packetQueue는 RecvIO 한 스레드만 push 한다 → 큐 안 순서가 보장된다. drain 권한은 CAS 로 항상 한 워커만 가진다 → 그 워커가 _zone.DoAsync를 부를 때 single producer 가 되어 채널 reorder가 일어나지 않는다. 다른 세션의 워커들은 자기 세션을 별도로 drain — 세션 간 병렬은 그대로.

Disconnect 도 같은 큐로

연결이 끊어지면 HandlePlayerDisconnect를 호출해야 하는데, 이걸 별도로 InboundCommands에 직접 넣으면 EnterZone/Move 등이 처리되기 전에 disconnect가 먼저 실행되는 경우가 생긴다. 그래서 sentinel 문자열 DisconnectMarker를 같은 _packetQueue 끝에 넣고, drain이 그것을 만나면 disconnect 로직을 실행한다. 모든 패킷과 disconnect가 동일한 순서 보장 트랙을 탄다.

8.5 그 외 다듬은 자잘한 것들

  • RecvLoop finally의 자체 정리 — 정상 종료 경로(Read==0)에서도 Dispose()를 호출하도록 변경. 자기 스레드 join 데드락은 Thread.CurrentThread != _recvThread 체크로 회피.
  • BlockingCollection은 Dispose하지 않음 — 워커가 PlayerActor 큐에 남은 잔여 패킷을 늦게 처리하면서 SendPacket을 호출하는 race가 흔하다. 큐가 disposed라면 ObjectDisposedException이 actor Flush 안에서 터진다. CompleteAdding만 하고 GC에 맡긴다.
  • EnqueueSend는 InvalidOperationException 흡수 — 위 race에 대비해 Add 호출의 모든 실패를 silently 무시. 송신 실패가 actor 크래시로 번지지 않도록.
  • pendingText 64KB 상한 — \n 없는 입력으로 StringBuilder가 무한 커지지 않도록 방어적 상한.
  • Accept loop의 종료 race — accept 직후 종료가 트리거되면 새 TcpClient를 즉시 닫고 break 해 leak 방지.
요점 · 동기 IO 모델은 단순해 보이지만, 멀티 워커가 actor 큐를 공유하는 순간 미묘한 ordering race들이 튀어나온다. "같은 도메인 객체에 대한 일은 같은 한 스레드가 순차로"라는 규칙을 세션 레벨까지 적용해야 안전하다.
